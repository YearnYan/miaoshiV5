## 1. 产品目标

### 1.1 目标描述
“秒视单词”旨在把传统背单词转化为高强度、可复玩的挑战体验：
- 通过极短曝光（默认 30ms）刺激快速识别能力。
- 通过连击、得分、排名形成持续挑战动机。
- 通过“免费额度 + 解锁”形成可持续商业闭环。

### 1.2 业务目标
- 提升碎片化学习场景下的单词识别效率。
- 建立“好友可比较、可挑战”的传播链路。
- 在不牺牲体验的前提下，实现付费转化。

### 1.3 成功判定（首版建议）
> 以下为可执行的目标阈值，若运营策略变化可在版本评审时调整。
- 新用户首日完成 1 局挑战率 >= 60%。
- 触发解锁页用户的支付或邀请行为率 >= 20%。
- 单用户 7 日内平均挑战局数 >= 5。

## 2. 范围定义

### 2.1 范围内（本期）
- 微信授权登录。
- 首页（开始挑战 / 好友排行 / 我的）。
- 游戏页（闪视、选择释义、反馈、连击、得分）。
- 结果页（评级、分数、新增词汇、再来一局、分享战绩）。
- 解锁拦截页（9.9 月卡 / 29.9 终身 / 邀请解锁 24h）。
- 好友榜与世界榜。
- 我的页面（用户信息、词汇量、VIP 状态、错题入口）。
- 词库拉取、缓存、版本校验。
- 订单创建、支付回调、权益发放。
- 核心行为埋点。

### 2.2 范围外（本期不做）
- 独立 App（iOS/Android 原生）。
- 复杂内容社区（评论、帖子、群组）。
- 管理后台大屏与 BI 分析系统。
- 多语言 UI（英语界面等）。

## 3. 目标用户

- 考试突击用户：四六级、考研、雅思、托福、GRE 备考人群。
- 竞技型用户：喜欢“极限挑战 + 排名对比”的学习者。
- 学生群体：有明确词汇目标，愿意用游戏化方式反复训练。
- 碎片时间用户：在通勤、排队、课间进行短时高频训练。

---

## 4. 核心用户场景

### 场景 1：快速挑战
用户打开小程序 -> 一键开始 -> 在高压节奏下完成一局 -> 获得评级与战绩。

### 场景 2：额度耗尽后解锁
用户累计达到免费额度上限 -> 弹出解锁页 -> 选择支付或邀请好友后继续训练。

### 场景 3：社交比较
用户查看好友排行 -> 发起挑战分享 -> 形成回流与复玩。

### 场景 4：个人追踪
用户进入“我的”查看词汇量、历史成绩、VIP 权益、错题记录。

## 5. 功能需求

### FR-01 登录与身份体系（P0）
- 触发条件：点击“开始挑战 / 好友排行 / 我的”。
- 输入：微信登录授权。
- 输出：用户身份（uid）、昵称、头像、权益状态。
- 业务规则：
  - 登录成功后才可写入成绩、词汇量、排行榜与订单。
  - 登录失败时保留重试入口，不允许直接进入正式对局。

### FR-02 首页入口（P0）
- 页面元素：
  - Logo（闪电符号）
  - 产品名“秒视单词”
  - 副标题“// flash_blitz”
  - 主按钮“$ start_challenge()”
  - 底部入口“好友排行”“我的”
- 业务规则：
  - 默认展示当前难度。
  - 默认展示系统在线状态。

### FR-03 闪视训练主循环（P0）
- 流程：
  1. `Flash`：中央展示英文词汇（按难度控制曝光时长）。
  2. `Input`：显示 4 个中文释义选项（1 正确 + 3 干扰）。
  3. `Feedback`：即时显示正误、连击变化与得分变化。
  4. `Next`：约 800ms 后进入下一词。
- 业务规则：
  - 默认难度为 Godlike（30ms）。
  - 前 10 题正确率 < 30% 时触发降级建议弹窗。
  - 用户可接受或拒绝降级建议。

### FR-04 难度系统（P0）
- 难度档位（建议）：
  - Godlike：30ms
  - Pro：80ms
  - Fast：150ms
  - Rookie：300ms
- 业务规则：
  - 新用户默认 Godlike。
  - 单局内支持暂停后继续，不重置难度。
  - 用户主动改档时实时生效于下一词。

### FR-05 词库与选项生成（P0）
- 输入：词库标识（如中高考/CET4/CET6/考研/雅思/GRE）。
- 输出：单词、释义、干扰项集合。
- 业务规则：
  - 词库支持本地缓存与版本更新。
  - 干扰项应尽量同词性、同难度层级，减少“秒猜”概率。

### FR-06 词汇量统计（P0）
- 定义：用户在练习中“首次答对”的词计入个人词汇量。
- 里程碑：10 / 100 / 1000 / 5000 / 10000。
- 业务规则：
  - 同一单词重复答对只计一次。
  - 到达里程碑时展示成就反馈。

### FR-07 计分与结果页（P0）
- 结果页输出：
  - 最终得分
  - 最大连击
  - 新增词汇数
  - 评级（如 A/S/SS）
- 业务规则：
  - 分数由“正误 + 连击 + 难度”共同决定。
  - 结果页提供“再来一局”“生成战绩海报”。

### FR-08 排行榜与挑战分享（P0）
- 好友榜：展示微信好友内成绩排序。
- 世界榜：展示全服 Top 50。
- 业务规则：
  - 支持从排行榜发起挑战分享卡片。
  - 排行数据以服务端复算成绩为准。

### FR-09 解锁拦截与付费（P0）
- 触发条件：达到免费额度上限。
- 解锁页文案核心：免费额度已耗尽 + 当前进度提示。
- 选项：
  - A：¥9.9 月卡（主推）
  - B：¥29.9 终身
  - C：邀请好友获得 24h 临时权限
- 业务规则：
  - 支付成功后立即发放权益并返回训练流。
  - 失败/取消需给出明确可重试提示。

### FR-10 邀请裂变（P0）
- 动作：用户分享小程序挑战卡。
- 生效：受邀用户首次有效进入后，邀请人获得对应词库 24h 权限。
- 业务规则：
  - 权限为时效性权益，过期自动回收。
  - 同一邀请关系防刷限频。

### FR-11 我的页面（P1）
- 展示内容：
  - 昵称、头像、当前等级
  - 累计词汇量
  - VIP 状态与到期时间
  - 历史最好成绩
  - 错题本入口
- 业务规则：
  - 权益状态与订单记录保持一致。

### FR-12 数据埋点（P0）
核心事件：
- `login_success`
- `game_start`
- `flash_answer`
- `round_finish`
- `paywall_show`
- `pay_click`
- `pay_success`
- `invite_send`
- `invite_success`
- `rank_view`

## 6. 关键状态与边界情况

- 加载态：词库加载中需可视化反馈，超时可重试。
- 空态：排行榜无数据时展示引导文案与挑战入口。
- 网络异常：保持已缓存词库可离线训练，联网后补上报。
- 中断恢复：来电/后台挂起后，回到前台可继续当前局。
- 支付中断：支付未完成不发放权益，支持订单状态补偿查询。
- 防刷分：结果分数以服务端复核为准，客户端仅作实时展示。

## 7. UI 设计硬约束（与参考图对齐）

### 7.1 风格关键词
- 终端感、极简、黑底高对比、霓虹绿主色。

### 7.2 色彩规范
- 背景主色：`#050806`
- 主强调色（霓虹绿）：`#1EDC74`
- 文本主色：`#E8FFE8`
- 次级文本：`#6E7A72`
- 辅助按钮底色：`#171A1F`
- 警示/排行强调：`#F5A623`
- 信息色：`#4A84FF`

### 7.3 字体规范
- 数字/代码风文本优先使用等宽字体风格。
- 主标题加粗，按钮文案采用“命令行语义”写法。

### 7.4 组件规范
- 主 CTA 按钮：高亮绿底，暗色文字，圆角中等，具轻微辉光。
- 次按钮：深灰底 + 对应语义色图标。
- 卡片：深色容器 + 细边框，避免高密度装饰。

### 7.5 动效规范
- 仅保留必要动效：闪词、答题反馈、按钮按压。
- 禁止花哨转场，避免分散注意力。

## 8. 验收标准

- AC-01：用户可完成“登录 -> 开始挑战 -> 结算 -> 再来一局”完整闭环。
- AC-02：默认 30ms 闪视可用，降级弹窗在规则触发时出现。
- AC-03：免费额度达到阈值后必然进入解锁页。
- AC-04：支付成功后 3 秒内权益生效并可继续训练。
- AC-05：邀请成功后可正确发放 24h 临时权限。
- AC-06：好友榜与世界榜均可正常展示与刷新。
- AC-07：UI 风格符合“黑底 + 霓虹绿终端风”，与参考图一致。

---

## 9. 非目标

- 不做复杂社交关系链（群聊、评论、私信）。
- 不做后台运营大盘。
- 不做多端统一账号体系（仅微信生态）。

## 10. 待确认项（含建议方案）

### 待确认项 1：免费额度口径
- 待确认原因：现有描述存在“全局 100 词”与“每词库 100 词”两种理解。
- 采用“全局累计 100 词”口径。

### 待确认项 2：是否允许未登录试玩
- 待确认原因：试玩可降门槛，但会削弱排名与转化数据闭环。
- 本期不开放匿名正式对局，统一先登录。

### 待确认项 3：月卡是否自动续费
- 待确认原因：影响支付流程、文案合规与用户预期。
- 默认不自动续费。

### 待确认项 4：邀请奖励生效条件
- 待确认原因：仅“点击进入”容易被刷，条件过严又影响裂变效率。
- 受邀用户“登录并完成 1 题”后发放奖励。

### 待确认项 5：世界榜刷新策略
- 待确认原因：实时刷新成本高，缓存过久影响体验。
- 服务端每 5 分钟更新一次榜单缓存。
